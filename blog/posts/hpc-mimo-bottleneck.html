Why GPU Memory Bandwidth Is the Real Bottleneck in MIMO Detection

Adel Dabah · Draft — 2026 · 8 min read

Large-scale MIMO detection is computationally intensive. When you move this workload onto a GPU, the first instinct is to focus on arithmetic throughput — FLOPs, tensor cores, mixed precision. But in practice, the real limiter is almost always memory bandwidth.

The Problem

A sphere decoder for a 64×64 MIMO system needs to traverse a tree of candidate vectors. Each node evaluation involves a matrix-vector multiply — which is memory-bound, not compute-bound, at this size. The A100 has 2 TB/s of HBM bandwidth, yet it still stalls waiting for data when the access pattern is scattered.

Low-Precision as a Bandwidth Trick

Switching from FP32 to FP16 halves the memory footprint of the channel matrix. At 64×64 complex entries, that drops from 32 KB to 16 KB — enough to fit in L2 for a batch of candidates. Our ISC 2023 paper showed this yields a 1.8× throughput improvement with negligible BER loss above 10 dB SNR.

The key insight: precision reduction is not primarily an arithmetic optimization — it is a caching optimization.

Practical Tips

  • Profile with ncu --metrics l1tex__t_bytes before optimizing compute.
  • Tile your channel matrix to match L1 (192 KB on A100).
  • Consider FP16 accumulation only for the partial distance — accumulate in FP32.
  • Coalesced reads matter more than vectorization at this problem size.

Code Sketch

// Load H in FP16, compute partial distance in FP32
__half H_fp16[N][M];
float  partial_dist = 0.0f;

for (int i = 0; i < M; ++i) {
    float val = __half2float(H_fp16[level][i]) * x[i];
    partial_dist += val * val;
}

This is a simplified sketch — the full implementation handles complex arithmetic and shared memory staging. See our ISC 2023 paper for details.
